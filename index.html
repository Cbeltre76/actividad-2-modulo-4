Actividad de Estimación de Proyectos de Software

1. Qué son los Modelos de estimación?

Los modelos de estimación son métodos y técnicas utilizados para predecir el esfuerzo, tiempo y costos necesarios para completar un proyecto de software. Estos modelos ayudan a los equipos de desarrollo a planificar y gestionar recursos de manera efectiva, proporcionando una base para la toma de decisiones y la planificación de proyectos.

2. Comparación de Modelos de Estimación: COCOMO y Puntos de Función

Modelo COCOMO

El modelo COCOMO (Constructive Cost Model) es un modelo de estimación basado en algoritmos desarrollado por Barry Boehm en 1981. COCOMO utiliza ecuaciones matemáticas para estimar el esfuerzo en función del tamaño del software medido en líneas de código (LOC) y otros factores de ajuste. Existen tres niveles de COCOMO: básico, intermedio y avanzado, cada uno aumentando en complejidad y precisión.

Método de Puntos de Función

El método de Puntos de Función es una técnica de estimación basada en la funcionalidad del software. Se centra en contar y clasificar las funcionalidades del sistema, tales como entradas, salidas, consultas, archivos y interfaces externas. Cada una de estas funcionalidades se pondera según su complejidad, y el total ponderado proporciona una medida del tamaño del software independiente de la tecnología utilizada.

Comparación

COCOMO es más adecuado cuando se dispone de datos históricos sobre proyectos anteriores y cuando el tamaño del software en líneas de código es una medida viable. Es útil para proyectos grandes y complejos donde los factores de ajuste pueden proporcionar una estimación más precisa. Por otro lado, el método de Puntos de Función es ideal para proyectos en los que se requiere una estimación temprana sin necesidad de conocer el detalle técnico. Es independiente del lenguaje de programación y la tecnología, lo que lo hace útil en fases iniciales del proyecto.

3. ¿A qué llamamos la estimación de un proyecto de software?

La estimación de un proyecto de software se refiere al proceso de predecir el esfuerzo, tiempo, costos y recursos necesarios para completar un proyecto de desarrollo de software. Esto incluye la identificación de todas las tareas y actividades requeridas, así como la asignación de recursos y la planificación del cronograma del proyecto.

4. Principios Esenciales para el Desarrollo de Sistemas

  • Modularidad: Descomposición del sistema en módulos o componentes más pequeños y manejables.
  • Cohesión: Grado en el que las tareas realizadas por un módulo son relacionadas y unificadas.
  • Acoplamiento: Medida en la que un módulo está interconectado con otros módulos.
  • Abstracción: Proceso de simplificar la complejidad mediante la reducción de los detalles innecesarios.

Ejemplo de Aplicación de la Modularidad

En el desarrollo de un sistema de gestión de inventario, la modularidad se podría aplicar dividiendo el sistema en módulos como gestión de productos, gestión de proveedores, gestión de pedidos y gestión de clientes. Cada módulo funcionaría de manera independiente pero integrándose al sistema global, facilitando el mantenimiento y la escalabilidad.

5. Lista de Partes Interesadas en un Proyecto de Desarrollo de Software para una Aplicación Móvil

  • Cliente: Define los requisitos y expectativas del proyecto.
  • Usuarios Finales: Utilizan la aplicación móvil y proporcionan retroalimentación.
  • Gerente de Proyecto: Planifica y supervisa el desarrollo del proyecto.
  • Desarrolladores: Implementan el código y desarrollan la aplicación.
  • Diseñadores UX/UI: Diseñan la interfaz de usuario y la experiencia del usuario.
  • Testers/QA: Verifican y validan que la aplicación funcione correctamente.
  • Equipo de Marketing: Promocionan la aplicación y gestionan la comunicación con los usuarios.

7. Normas y Estándares en el Desarrollo de Software

Para garantizar la consistencia en la codificación y la documentación en el desarrollo de software, se deben seguir normas y estándares como:

  • Convenciones de Nombres: Usar nombres significativos y consistentes para variables, funciones y clases. Ejemplo: Usar camelCase para nombres de variables y PascalCase para nombres de clases.
  • Documentación: Documentar el código con comentarios claros y mantener un manual del usuario y una guía de desarrollo. Ejemplo: Usar comentarios Javadoc para describir las funciones y sus parámetros en Java.
  • Control de Versiones: Usar sistemas de control de versiones como Git para gestionar los cambios en el código. Ejemplo: Realizar commits frecuentes y utilizar ramas para el desarrollo de nuevas características.
  • Pruebas: Implementar pruebas unitarias, de integración y de sistema para asegurar la calidad del software. Ejemplo: Usar frameworks de pruebas como JUnit para realizar pruebas unitarias en Java.
  • Revisiones de Código: Realizar revisiones de código periódicas para detectar y corregir errores. Ejemplo: Utilizar herramientas como GitHub para realizar pull requests y revisiones de código colaborativas.
